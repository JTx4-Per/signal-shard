2 · Reducer Priority Resolver deterministic signal ordering — prevents oscillation

Why ordering matters

A single inbound email can carry "thanks" + a new ask. Without a fixed tier, rules and model oscillate. Tier ① short-circuits tier ②, etc.

Soft vs hard complete

soft_complete (model-inferred) is reversible — a later message can reopen. hard_complete (explicit "done/closed/signed") is sticky and only reopens on a new ask, not on FYI follow-ups.

Sent Items is mandatory

Tier ③ depends on Sent Items being synced in the same delta window. Without it, respond_open → waiting_on never fires and the Respond list grows unboundedly.

Bounded time window

Reducer input is the snapshot [conversation_start … latest_event_timestamp]. Tier ③ requires sent.ts ≤ latest_inbound.ts — otherwise out-of-order delta pages could flip state incorrectly. "Now" is passed as a parameter, never read inside the reducer.

Determinism

Same snapshot + same "now" → same transition. Required for event-log replay and regression tests.

7 · Hard Invariants enforce in code, not just docs

I1 · Reducer is sole task author

Classification never writes tasks. Only the reducer, after priority resolution + idempotency check, emits writeback intents. Classifier outputs are evidence, not decisions.

I2 · One active task per conversation

unique(conversation_id) where todo_tasks.status in (notStarted, inProgress). Dedupe key = hash(conversation_id + task_kind). Two-way mapping stored both directions.

I3 · Sent Items must be in the snapshot

Reducer refuses to transition respond_open → waiting_on unless the snapshot window includes Sent Items up to latest_received_at.

I4 · Priority tiers are total-ordered

Resolution (①) short-circuits every lower tier. Ties within a tier fall through; tiers never merge. No "weighted score" blending.

I5 · Soft complete is reversible for N days

A later inbound message within the soft window can revert to updated. Hard complete is sticky — only a new ask (tier ②) reopens it.

I6 · Every writeback is idempotent

Writeback carries an operation_key. Second attempt with the same key is a no-op. Retries are safe; webhook storms cannot duplicate.

I7 · needs_review blocks writeback

State-level needs_review suspends all Graph writes for the conversation until resolved (manual or by a later disambiguating message).

I8 · conversation_events is append-only

State is derivable by replaying events. Destructive edits forbidden; corrections are compensating events.

I9 · Reducer is deterministic

Same inputs → same transition, always. No wall-clock reads, no RNG, no non-idempotent side calls inside the reducer. "Now" is passed in as a parameter. Required for replay, debugging, and trust.

I10 · Reducer operates on a bounded time window

Snapshot window = [conversation_start … latest_event_timestamp]. Transition respond_open → waiting_on requires sent_items.timestamp ≤ latest_inbound.timestamp. Out-of-order Graph deltas must not flip state; late messages replay through the reducer.

I11 · Handoff evidence is narrow

delegate → waiting_on requires outbound email with forward/assignment language or explicit mention of a new owner. Model hint alone is insufficient — prevents hallucinated delegations.

F1 · Future: multi-action threads (don't block structurally)

v1 enforces one active task per conversation. Schema should already support multiple: prefer todo_tasks.conversation_id without a hard single-row constraint — use the "unique-when-active + action_slot" shape so v2 can decompose "send deck + confirm timeline" into 2 tasks without migration pain.
