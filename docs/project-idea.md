Recommended architecture
1) Source of truth

Use the mailbox as the canonical source for raw messages, and your own database as the canonical source for computed state.

A good minimal schema is:

messages
message_id
internet_message_id
conversation_id
folder
from, to, cc
subject
received_at
body_text
categories
is_read
has_attachments
importance
classifications
message_id
primary_bucket
flags_json
reason_short
confidence
model_version
classified_at
tasks
message_id
todo_task_id
task_status
due_at
waiting_on
threads
conversation_id
thread_state
last_actionable_message_id

Why thread state matters: most email action is really at the thread level, not the message level. Without a thread-aware layer, the system will overcreate tasks.

2) Ingestion

Start with delta query on the Inbox and Sent Items, not full rescans. Delta is specifically intended to track created, updated, and deleted items incrementally. For freshness, you can optionally pair that with Graph change notifications and then reconcile with delta so you do not miss events. Microsoft documents both patterns, and also documents lifecycle notifications to reduce missed subscription gaps.

A practical pattern:

Initial backfill: last 90 to 180 days.
Ongoing:
webhook receives “mail changed”
enqueue mailbox/folder sync
run delta query
fetch only changed messages
reclassify affected threads
3) Normalize the message for classification

Before any model call, convert the email into a compact structured record:

sender role signals: boss, direct report, external vendor, mailing list, automated
recipient position: am I in To, Cc, or Bcc
reply-state: did I send the last email in this thread
thread age and latest activity
meeting/date signals
imperative/action phrases
explicit asks
deadlines
attachment mentions
unsubscribe/list headers
newsletters/promotions/autogenerated indicators

Graph’s message resource exposes properties like recipients, categories, conversation metadata, read state, and supports retrieving message content. Microsoft Learn also documents using the Prefer: outlook.body-content-type="text" header for text bodies in GET scenarios where supported, which is helpful for classifier preprocessing.

4) Hybrid classifier, not LLM-only

For this use case, best results usually come from a rules + model system:

Deterministic layer first
newsletters, noreply, list mail, promos → Delete/Unsubscribe or FYI
receipts, confirmations, system alerts → FYI unless they imply action
“can you”, “please send”, “need by”, “action required” → Act or Respond
you are only cc’d and no direct ask → FYI by default
last sender is you and no reply yet from recipient → Waiting-on
explicit “let X handle this” or forwarded request → Delegate
Model second
resolve ambiguous cases
extract action owner, due date, waiting-on entity, urgency, and rationale
classify at message level and then fold into thread state

This is much more stable than asking one model prompt to do everything.

5) Thread resolution logic

A message can say “Thanks, done” and should close an earlier Act. So the system needs thread reducers such as:

if newest message confirms completion → close prior Act task
if newest message is from you and asks something → Waiting-on
if newest message from other party asks you something → Respond or Act
if task exists and new mail changes deadline → update task
if later message makes earlier ask irrelevant → archive the task

This is the layer that makes the system feel intelligent instead of just “email tagging.”

How I’d define the buckets operationally
Act

Use when the email implies a deliverable or non-trivial work item.

Examples:

build the deck
review the contract
send numbers by Friday
update the doc
investigate the bug

Create a To Do task here nearly always.

Respond

Use when the main obligation is communication, not production work.

Examples:

answer a question
confirm attendance
send a short yes/no
provide a quick opinion

Also a To Do candidate, but often lower-effort.

Delegate

For personal use, this should mean:

I need to hand this to someone else or track that someone else owns it.

Since you said personal use only, I would not auto-assign anything externally. I would:

create a To Do task for you,
track owner_expected = other_person,
optionally draft a forwarding/reply suggestion,
switch to Waiting-on after you send the delegation.
Defer / Waiting-on

This should have two subtypes:

defer_until_date
waiting_on_person

That distinction matters because the UX is different:

defer goes into a snooze queue
waiting-on goes into a follow-up queue
FYI / Archive

No task, no reminder. Possibly tag for future search.

Delete / Unsubscribe

No task. Optionally queue unsubscribe suggestions when list-style signals are strong.

Escalate

Make this a flag:

needs_escalation = true
escalation_type = urgency | authority | risk

That avoids turning your taxonomy into a confusing overlap mess.

To Do integration design

Microsoft Graph’s To Do APIs support task lists and tasks, and a todoTask can include linkedResource data to point back to the source item. That makes it ideal for “action extracted from email” workflows. To Do also has built-in lists plus custom lists you can create.

I would create exactly 4 task lists:

Email - Act
Email - Respond
Email - Waiting On
Email - Delegated

Not one list per sender or project. Keep it simple.

Task title examples:

Reply to Sarah about Q2 budget
Draft proposal for vendor renewal
Waiting on Tom for signed contract

Task notes/body:

short model reason
extracted due date
message link / webLink / identifiers
latest sender and date

Task sync rules:

create when actionable item first appears
update if subject/action changes materially
complete when thread resolves
dedupe per thread, not per message
Where to store your classifier state

You have three viable options:

Option A: local DB only

Best if you want maximum control and easy analytics.

Option B: Outlook categories + local DB

Graph message objects include categories, so you can stamp user-visible Outlook categories like AI-Act, AI-Respond, AI-Waiting. That gives you a visible layer inside Outlook while still keeping richer state locally.

Option C: Graph extensions + local DB

Graph supports open extensions and Outlook extended properties on messages. For a personal system, this is useful for storing your app’s hidden metadata such as classifier version or task link state. Microsoft’s docs explicitly describe extensions for app-specific data, and for Outlook resources they recommend open extensions for many common scenarios.

My recommendation:

use categories for human-facing labels
use your DB for main logic
optionally use open extensions for small mailbox-resident metadata
A strong classification output shape

Instead of asking a model for just one label, have it output JSON like:

{
  "primary_bucket": "Respond",
  "secondary_flags": {

  "action_owner": "me",

  "confidence": 0.89

That gives you decisionability, auditability, and easier tuning.

Critical signals that improve quality a lot

These are the highest-value features:

Am I in To vs Cc
Did I send the last message
Is there a direct ask
Is there a due date
Is the sender a person vs system
Is this a mailing list or bulk message
Does this thread already have an open task
Did a later message already resolve the earlier ask
Was the email addressed to a group alias instead of me
Did I already reply from Sent Items

That last one is especially important: if you ingest Sent Items too, you can distinguish “owes response” from “already responded, now waiting.”

Personal-use privacy and safety posture

For personal use, I would strongly avoid:

auto-sending replies
auto-forwarding/delegating
auto-deleting mail except obvious bulk-mail folders
auto-unsubscribing without review

I would allow:

auto-tagging
auto-task creation
auto-task completion when confidence is high
daily summary digest
suggested drafts, not sent drafts

That keeps the system useful without becoming dangerous.

Example decision policy

A clean first version could be:

Delete/Unsubscribe
bulk mail or system noise with no personal relevance
else FYI
no direct ask, informational only
else Respond
communication task under 10 minutes, no substantive deliverable
else Act
deliverable/research/review/production work
Delegate only if there is explicit evidence another person should own it
Waiting-on
if latest blocking state is external, or you already sent the requested next step

This policy reduces overuse of Delegate and Waiting-on.

My recommendation on “Delegate”

Because this is personal-use only, I would not make Delegate a top-level automation target at first. I would model it as:

primary bucket = Act or Respond
flag = candidate_delegate

Then, after you actually forward or assign it, transition it to Waiting-on.

That mirrors real workflow better and avoids phantom delegated tasks.
